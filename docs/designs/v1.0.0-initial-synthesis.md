# Build plan: `webdesign` — the web design + art-direction + build skill

> **Implemented in: v1.0.0** (2026-07-15) — the initial release's build plan/design record, grounded
> in [../research/provenance.md](../research/provenance.md) · [CHANGELOG](../../CHANGELOG.md).

**Goal.** Package an end-to-end web design, art-direction, and frontend-build system as a
standalone Agent Skill, **`webdesign`** (`~/.claude/skills/webdesign/`, invoked as `/webdesign`):
one router file plus a library of copy-pasteable craft references, distilled from a production
website-rebuild pipeline's design corpus rather than written from scratch.

## 1. Why router-over-references, not one long document

The corpus this skill is built from — a full pipeline covering direction, structure, per-section
craft, build, and two review gates — runs to roughly 4.5k lines across 8 skills and 4 agent briefs
once pipeline machinery (run directories, phase codes, CMS wiring) is stripped out. Inlining that
into a single skill file would either force every invocation to read all of it, or force brittle
manual excerpting per task.

Instead, `SKILL.md` stays a **thin router**: the six-stage arc, a pick-by-job table, and the
non-negotiable laws, at **152 lines** — small enough that an agent reads the whole thing on every
invocation for free. Everything else — the actual craft (layout, typography, color, depth,
decoration, imagery, motion, experimental technique, per-vertical strategy, persuasion, trust) and
the process documents (direction authoring, structure/IA, the Tailwind bridge, the build patterns,
art review, scored critique) — lives in `references/`, loaded only when the current job needs it.
This is the same progressive-disclosure shape as the source pipeline's own skill layering: a router
plus on-demand references beats one document trying to serve every job at once.

## 2. The six-stage arc, not nine strategies

Where a dispatch-style orchestration skill is naturally a **catalog of interchangeable strategies**
(pick one, run it), a design skill is naturally an **arc with a fixed shape but flexible entry
points**: direction → structure → per-section craft → build → art review → critique/QA gate. The
stages aren't presets over shared dimensions the way orchestration strategies are — each stage does
a genuinely different *kind* of work (authoring a contract vs. laying out an IA vs. shipping CSS vs.
scaffolding a Next.js app vs. two distinct review disciplines). What carries over from that other
shape is the *discipline* around entry and exit: any stage can be entered standalone (stage 3 for
one section, stage 5 for a build you didn't make), and stages 5–6 are a genuine two-gate system —
taste review, then a mechanical/scored gate — rather than one review pass wearing two hats.

## 3. How v1.0.0 was built

Synthesized 2026-07-14 from a production website-rebuild pipeline's design corpus — 8 skills plus 4
agent briefs, roughly 4.5k lines of source material — via a staged multi-agent run:

1. **Three parallel authors**, each porting one slice of the corpus (the craft references, the
   direction/build process docs, the review/critique gates) into `webdesign`-shaped reference files
   — restyled from pipeline-specific prose into standalone, project-agnostic craft documents.
2. **An opus synthesizer** wrote the router (`SKILL.md`) from all three authors' output: the
   six-stage arc, the pick-by-job table, and the seven non-negotiable laws, at the 152-line budget.
3. **A four-way adversarial review** — port fidelity byte-diff-verified for 9 craft files (checking
   that technique content ported without silent drift), plus three further review passes on
   structure, cross-links, and attribution. Result: zero blocker/major findings; 5 minor
   cross-link/attribution fixes applied before release.

Pipeline coupling — run directories, phase codes (P0–P6), CMS-specific wiring, source-pipeline-
specific terminology — was stripped throughout; every reference reads as a standalone technique
document, not a stage in someone else's pipeline. External citations (the design-writing and
CSS-technique sources each reference draws on) were consolidated into a single `SOURCES.md` rather
than repeated per file, so citation lists don't cost tokens on every on-demand read.

## 4. What's out of scope for v1.0.0

- **No behavior/props engineering for reusable components** — that's `building-components`; this
  skill composes *sections* from a component layer, it doesn't build the layer.
- **No asset generation** — `generate-image`/`generate-video` produce the images/video a direction
  calls for; this skill decides what a slot needs.
- **No full accessibility audit** — stage 6 does a design-level contrast/focus pass; a WCAG 2.2 deep
  audit is `accessibility`.
- **No orchestration/dispatch machinery** — no run ledger, no config file, no multi-agent fan-out.
  Each stage is something one agent (or a human) does directly; nothing here needed the dispatch
  layer the template repo this project is structured after was built for.
