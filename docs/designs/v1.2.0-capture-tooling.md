# Design: capture-tooling contract for the visual stages — v1.2.0

> **Implemented in: v1.2.0** (2026-07-16, tag `v1.2.0`) — grounded in
> [../research/visual-capture-tooling.md](../research/visual-capture-tooling.md) ·
> [CHANGELOG](../../CHANGELOG.md).

## Goal & non-negotiable constraint

Name the review stages' real prerequisite — a serve + screenshot + vision path — and recommend a
token-efficient toolchain, **without adding runtime machinery**. The skill stays "a router plus
references, no scripts, no config"; what ships is a stated *contract* and a recommendation
*ladder*, not a probe script or a hard dependency.

## Design principles

1. **Contract over tool mandate** (the v1.1.0 stack-posture doctrine applied to tooling): the
   normative part is the capability contract — serve · per-section crops at two viewports with
   reduced motion · a model that reads images. The named tools are recommendations with a
   fallback ladder.
2. **Match the tool to the job's token shape.** Batch capture is deterministic → script it (zero
   tokens while running). Exploration is interactive → `agent-browser` (terse CDP CLI, no
   Playwright dependency, self-documenting). Host-native browser tools are the fallback.
3. **Degrade honestly, never silently.** With no browser, the review becomes a code-level pass
   that must label itself "not an art review; no build was viewed."
4. **One statement, referenced everywhere it's needed:** the contract lives in `art-review.md`;
   the build self-check, SKILL.md stage 5, and the installation prerequisites point at it rather
   than restating it.

## Considered and rejected

Recorded in the research doc: shipping a capture script (hard-codes one stack's markup; violates
no-machinery), a `toolbox`-style probe script (same reason), and mandating a single tool
(contradicts the stack posture). See
[../research/visual-capture-tooling.md](../research/visual-capture-tooling.md).
