# Design: scope rule + stack-agnostic build posture — v1.1.0

> **Implemented in: v1.1.0** (2026-07-15, tag `v1.1.0`) — grounded in
> [../research/stack-agnosticism.md](../research/stack-agnosticism.md) ·
> [CHANGELOG](../../CHANGELOG.md).

## Goal & non-negotiable constraint

Make the skill's technology posture explicit — Next.js + Tailwind v4 is the **worked example, not
a requirement** — and state the skill's scope boundary, **without touching the craft corpus**. The
audit showed ~85% of the skill was already stack-agnostic; the problem was framing (three
router sentences that made it *sound* Next-locked) plus one file that presented its reference
stack as a mandate. So this release is a reframe, not a rewrite: zero design-technique content
changed, and the 13 craft references have a clean diff.

## Design principles

1. **The patterns are the contract; the stack is the example.** Concrete beats abstract — the
   skill's founding principle ("every reference is copy-pasteable") forbids replacing the worked
   example with framework-neutral pseudo-guidance. Instead, every stack-specific mechanism gets a
   named portable equivalent.
2. **A project's existing stack always wins over the example.** Stated verbatim in both SKILL.md
   and `frontend-build-patterns.md`, so an agent joining an Astro/SvelteKit/Nuxt codebase never
   "helpfully" scaffolds Next.js.
3. **The scope rule is the inclusion test.** The skill owns everything between "what should this
   look like" and "the pixels on screen match it"; engineering that doesn't change what the user
   sees (build tooling, state, data fetching, testing, component-API design) is out of scope and
   hands off to the named complementary skills.
4. **Demote, don't delete.** The TypeScript/JavaScript style guide fails the inclusion test but is
   harmless and occasionally useful — it became an explicitly-labeled appendix ("engineering
   craft, not design doctrine; nothing in this appendix changes a pixel"), content byte-identical.
   Deleting it would be churn without benefit; if it never earns its keep, drop it in a major.
5. **One framing, every surface.** SKILL.md, README, docs/usage, docs/stages, the site (meta,
   hero, arc card, pick-by-job table), and llms.txt all say the same thing — no surface left
   claiming an unqualified "Next.js build".

## What changed (by file)

- **`skills/webdesign/SKILL.md`** — new **Scope / Stack posture** paragraph (the inclusion test +
  worked-example framing); frontmatter description, workflow-arc stage 4, and the pick-by-job row
  reworded ("Build the frontend from a design (any stack; Next.js worked example)"). Stays under
  the 200-line check-sync budget (163 lines).
- **`references/frontend-build-patterns.md`** — new **"Stack posture — the patterns are the
  contract, the stack is the example"** section with a mapping table: each load-bearing pattern
  (one component per section key · fail-loud registry · tokens→styling layer · static routes from
  content · optimized image/font primitives · redirects + not-found fallback) → its reference
  implementation → what to look for in your stack (Astro/Svelte/Vue/HTMX equivalents, StyleX /
  vanilla custom properties / CSS Modules). `## Stack` retitled `## Reference stack (the worked
  example)`. TS/JS craft demoted to a labeled appendix.
- **`references/tailwind-v4-theme.md`** — a "No Tailwind?" note: a `DESIGN.md` token export is
  just CSS custom properties; only the namespace mapping is Tailwind-specific.
- **README / docs/usage.md / docs/stages.md / site / llms.txt** — aligned to the same framing.

## Considered and rejected

- **`stacks/` adapter folder** (a stack-neutral core + `stacks/nextjs.md`, `stacks/astro.md`, …) —
  deferred, not rejected: premature with one worked stack; the mapping table is each future
  adapter's seed. (Research option B.)
- **Fully framework-neutral rewrite** — rejected; abstract build guidance is worse guidance.
  (Research option C.)
- **Deleting the TS/JS style guide** — rejected in favor of the labeled appendix (principle 4).
- **An exhaustive framework catalog** — rejected; examples are named as a dated snapshot with an
  explicit "heat shifts" disclaimer so staleness is never a correctness bug.
